01
Month 1–2 · 8 Weeks

DSA Sharpening

Goal: 100–120 problems solvedDaily: 1 hr DSA + 30 min reviewPlatform: LeetCode + Striver
Week-by-week breakdown
WEEK 01Arrays & Strings — The Foundation
▲
  • Arrays: two sum, best time to buy/sell stock, contains duplicate, product of array except self, maximum subarray (Kadane's)
  • Strings: valid anagram, group anagrams, longest substring without repeating characters
  • Prefix sum pattern: subarray sum equals K, range sum query, count subarrays with given sum
  • Sliding window intro: max sum subarray of size K, minimum window substring
  • LeetCode target: 12–15 problems. All Easy first, attempt 2–3 Medium by end of week
  • Resource: Striver SDE Sheet Day 1–4, NeetCode Arrays + Sliding Window playlist
WEEK 02Hashing, Two Pointers & Binary Search
▼
  • HashMap patterns: two sum (hashmap version), 4sum, count pairs with given sum, longest consecutive sequence
  • Two pointers: 3sum, container with most water, trapping rain water, valid palindrome
  • Sorting: merge intervals, meeting rooms I/II, sort colors (Dutch national flag)
  • Binary search: search in rotated sorted array, find minimum in rotated array, search a 2D matrix
  • LeetCode target: 12–15 problems. Mix Easy + Medium. "Trapping Rain Water" is a must-do
  • Resource: Striver Day 5–8, NeetCode Two Pointers + Binary Search
WEEK 03Linked Lists & Stacks/Queues
▼
  • Linked list: reverse LL, find middle, detect cycle (Floyd's), merge two sorted lists, remove Nth from end
  • Advanced LL: merge K sorted lists, LRU Cache (LC 146) — this is also a system design concept!
  • Stack: valid parentheses, min stack, daily temperatures, next greater element, evaluate reverse Polish
  • Monotonic stack: largest rectangle in histogram, next greater element II, asteroid collision
  • Queue: sliding window maximum, design circular queue, implement stack using queues
  • LeetCode target: 12 problems. LRU Cache is a favourite — interviewers love it
WEEK 04Trees — BFS, DFS, and BST
▼
  • Tree traversals: inorder, preorder, postorder — both recursive AND iterative (interviewers ask iterative)
  • BFS: level order traversal, right side view, zigzag traversal, average of levels
  • DFS: max depth, diameter of tree, path sum I/II, lowest common ancestor
  • BST: validate BST, kth smallest element, inorder successor, insert/delete in BST
  • Construct binary tree from preorder + inorder, serialize/deserialize binary tree
  • LeetCode target: 12–15 problems. Trees appear in ~30% of all interview coding rounds
  • Resource: Striver Tree series (watch fully), NeetCode Trees playlist
WEEK 05Heaps, Priority Queue & Backtracking
▼
  • Heap basics: min heap vs max heap in Java (PriorityQueue), top K frequent elements, K closest points
  • Merge K sorted arrays/lists using heap, find median from data stream (hard but important)
  • Recursion fundamentals: permutations, subsets, power set — understand the decision tree
  • Backtracking: N-queens, Sudoku solver, combination sum I/II, word search, palindrome partitioning
  • LeetCode target: 10–12 problems. Heaps are underrated — Flipkart and Swiggy both ask them
WEEK 06Graphs — BFS, DFS, Topological Sort, Union-Find
▼
  • Graph representation: adjacency list vs matrix — always use adjacency list unless told otherwise
  • BFS: shortest path in unweighted graph, word ladder, rotten oranges, 01 matrix
  • DFS: number of islands, clone graph, pacific atlantic water flow, max area of island
  • Topological sort: course schedule I/II (both BFS Kahn's and DFS approach)
  • Union-Find (DSU): redundant connection, number of provinces, accounts merge
  • Dijkstra: network delay time, cheapest flights within K stops, path with minimum effort
  • LeetCode target: 12 problems. Google and Uber love graph problems — don't skip this
  • Resource: Striver Graph series (absolute must-watch), NeetCode Graphs
WEEK 07Dynamic Programming — Part 1
▼
  • DP fundamentals: memoization vs tabulation, how to spot a DP problem (overlapping subproblems + optimal substructure)
  • 1D DP: climbing stairs, house robber I/II, coin change, decode ways, jump game
  • 2D DP: unique paths, minimum path sum, edit distance, maximal square
  • String DP: longest common subsequence, longest palindromic subsequence, palindromic substrings
  • LeetCode target: 12–15 problems. DP is the hardest topic — start Week 7 not Week 12!
  • Tip: for every DP problem, write the recurrence relation before coding
WEEK 08DP Part 2 + Trie + Revision Sprint
▼
  • Knapsack DP: 0/1 knapsack, subset sum, partition equal subset sum, target sum
  • Interval DP: burst balloons, remove boxes (Google level — try it)
  • Trie: implement trie from scratch, word search II, design add and search words
  • Full revision sprint: re-solve 1 hard problem from each topic. Time yourself — 25 mins max
  • Attempt 2 LeetCode weekly contests — get comfortable with the pressure format
  • Close at 100+ problems solved total. Start solving 1 Medium every morning — habit from here on

Month 4–5 · 6 Weeks

System Design (HLD)

Goal: 12 full system designs, each drawn + verbally practicedDaily: 30 min DSA + 1 hr system design
First — learn these building blocks (Weeks 20–21)

🧱Core concepts — know these cold

  • Horizontal vs vertical scaling — when to do each
  • Load balancers: L4 vs L7, round robin, least connections, consistent hashing
  • CAP theorem — you can't have all 3, give examples of CP and AP systems
  • SQL vs NoSQL — be able to choose and justify in 30 seconds
  • Caching: cache-aside, write-through, write-back, TTL, eviction policies (LRU, LFU)
  • Database replication: master-slave, read replicas, replication lag problem
  • Database sharding: range-based, hash-based, directory-based — and resharding problem
  • Message queues (RabbitMQ) vs event streaming (Kafka) — pull vs push, retention
  • CDN: what it caches, push vs pull, cache invalidation
  • Rate limiting algorithms: token bucket, leaky bucket, sliding window counter
  • Consistent hashing — how it solves the resharding problem
  • Bloom filters — probabilistic, no false negatives, space-efficient

📐The SD interview framework — use every time

  • Step 1 (5 min) — Clarify: Functional requirements (what features?), Non-functional (scale? latency? availability?)
  • Step 2 (3 min) — Estimate: DAU, QPS (queries/sec), storage per year. Write numbers on the whiteboard
  • Step 3 (5 min) — HLD: Draw the top-level boxes and arrows — don't go deep yet
  • Step 4 (15 min) — Deep dive: Interviewer picks 1–2 components, you go deep. DB schema, API contracts, cache strategy
  • Step 5 (5 min) — Bottlenecks: Where does this break? How do you fix it?
  • Always explicitly state: which DB and why, which cache and why, which queue and why
  • Always ask: "Should I focus on any particular component first?"
  • Use paperdraw.dev to draw WHILE talking — not after. Show thinking in real time

✏️ Exact diagram to draw: Client → API Gateway → App Server → check Redis (cache hit → return short URL). Cache miss → MySQL → Base62 encode → store in MySQL + Redis → return. Redirect flow: Client → CDN → App Server → Redis lookup → 301 redirect. Show cache-aside arrows clearly. Add analytics async write to separate DB via message queue.
Basic2. Rate Limiter
API GatewayRedis countersToken bucket algoDistributed lockConfig service
✏️ Exact diagram to draw: Token bucket diagram — bucket fills at rate R tokens/sec, each request takes 1 token, reject (429) when empty. Then distributed version: multiple app servers sharing Redis counter per user_id. Draw the race condition: two servers read counter=9 simultaneously → both allow → counter becomes 10 not 11. Fix: Redis Lua script for atomic check-and-increment.
Basic3. Key-Value Store (like Redis/Dynamo)
Consistent hashing ringPartitioningReplication (N=3)Quorum (W=2, R=2)Gossip protocol
✏️ Exact diagram to draw: Consistent hash ring with 4 server nodes. Show a key K hashing to a position on the ring, mapped to nearest clockwise node. Show virtual nodes (each server has 3 virtual nodes for even distribution). Draw 3-replica quorum write: coordinator receives write → sends to 3 nodes → waits for 2 ACKs (W=2) → returns success. Show what happens if 1 node is down.

Medium9. Payment System (Razorpay — very likely if you apply there)
Payment APIIdempotency layerFraud detectionBank APILedger DBRetry queue (Kafka)Reconciliation service
✏️ Exact diagram to draw: Client sends POST /payment with idempotency_key → check Redis idempotency store → if key exists, return cached response (prevents double charge). New request: Fraud detection service (ML model, sync check, <100ms SLA) → if pass → call Bank API → if success → double-entry ledger write (debit merchant_id, credit bank account — ACID transaction). Failed payments: Kafka dead letter queue → retry with exponential backoff (1s, 2s, 4s, max 5 retries). Draw reconciliation service separately — nightly batch job comparing internal ledger with bank statements.
Hard10. Google Drive / Dropbox
Block ServerS3Delta syncMetadata DBSync ServiceCDNNotification service
✏️ Exact diagram to draw: Client splits file into 4MB blocks, hash each block (SHA-256). Upload: only send blocks with new hashes (delta sync). Block Server: check if hash already exists in S3 → if yes, skip upload (deduplication). Metadata DB stores: file_id, user_id, file_name, block_hashes[] (ordered), version, updated_at. Conflict: two users edit same file offline → on sync, compare last_known_version → if mismatch, create conflict copy. Show multi-device sync: change on device A → Notification Service → push to all device clients → they pull delta.
Hard11. Web Crawler (Google)
URL FrontierDownloader clusterHTML ParserBloom filterDNS cacheURL deduplicationStorage
✏️ Exact diagram to draw: URL Frontier (priority queue, prioritised by PageRank + recency) → URL Dispatcher → Downloader Workers (horizontal scale) → HTML Parser → extract all URLs → Bloom filter (already crawled?) → if new → add to frontier. Show politeness policy box: separate sub-queue per domain, minimum 1s between requests to same host (respect robots.txt). DNS cache box: local cache reduces DNS lookups from 300ms to 2ms. Storage: raw HTML → S3. Parsed content → Elasticsearch for full-text search.
Hard12. Typeahead / Search Autocomplete (Google Search bar)
Trie serversQuery aggregation pipelineRedis (top-K per prefix)CDN (prefix cache)Data collection serviceZookeeper
✏️ Exact diagram to draw: Real-time path: Client types "swi" → CDN (cache frequent prefixes) → Trie server cluster → return top-5 suggestions sorted by frequency. Trie node stores: prefix string + top-5 suggestions list (so you don't traverse deep). Offline aggregation (left side): User query logs → Kafka → Aggregation workers (count frequencies) → weekly MapReduce job → rebuild trie → push new trie to servers via Zookeeper coordination. Show how "swiggy" gets to top for "swi" — it's the highest-frequency completion in the logs.

LLD — critical for Flipkart, CRED, Thoughtworks

🏗️LLD problems to solve (in order)

  • Parking lot system — most common, start here. Practice inheritance for vehicle types
  • Library management system — books, members, issued books, overdue fines
  • Hotel booking system — rooms, reservations, date range conflicts
  • Chess game — pieces, move validation, check detection
  • ATM machine — states: idle, card inserted, PIN verified, transaction
  • Elevator system — scheduling algorithms, multiple elevators
  • Snake and ladder board game — dice, players, power-ups
  • Food ordering system (mini Swiggy) — restaurants, menus, cart, order

📐Design patterns to know cold

  • Factory: PizzaFactory creates Margherita, BBQ, etc. without exposing class names
  • Singleton: Database connection pool — only one instance, thread-safe
  • Observer: Order status change → notify customer, restaurant, driver
  • Strategy: Payment: UPI/Card/NetBanking — swap algorithm at runtime
  • Decorator: Add toppings to pizza without changing base class
  • Command: Undo/redo in text editor — encapsulate actions as objects
  • SOLID principles — explain each with a real Spring Boot example
Behavioural — prepare these specific stories
Tell me about a time you resolved a production incident. Use your Amazon support experience directly. Even as support eng, you've diagnosed incidents. Frame it as: situation (what broke, business impact) → investigation (how you approached it) → resolution (what you did) → prevention (what changed after). Show systematic thinking.
Tell me about your most challenging technical problem. Pick the hardest bug you debugged at Amazon. Show your mental process: check logs → form hypothesis → test hypothesis → isolate variable → fix → validate → add monitoring so it never happens again.
Why are you moving from support to SDE? Be confident: "As a support engineer at Amazon, I've been closer to production systems than most SDEs at smaller companies. I've seen exactly how systems fail at scale. Now I want to build those systems — this 6-month preparation is proof of that intent, not just a claim." Then mention your 3 projects.
Tell me about a project you built that you're proud of. Use your Order Service with Kafka. Walk through decisions: "I chose Kafka over RabbitMQ because I needed message replay and audit trail. I implemented the Saga pattern because distributed transactions across 3 services can fail at any point..." This is SDE-level thinking from concrete experience.
Application strategy — when and how

🚀The application timeline

  • Month 5 (start applying Safe tier): Zoho, Thoughtworks — use these as warm-up. Real interviews reduce nerves more than 10 mock interviews
  • Month 6 (Target tier): Flipkart, Swiggy, Razorpay, CRED — your strongest applications. You're fully prepared here
  • Month 7 (Dream tier): Google, Microsoft, Uber — you're sharpest now with real interview experience from earlier rounds
  • LinkedIn: update "Open to Work", add all 3 project links, add AWS Educate badges, list technologies clearly
  • Referrals: your Amazon network is a goldmine. Look for ex-colleagues at Flipkart/Swiggy/Razorpay on LinkedIn — a referral 2x your resume shortlist rate
  • Before every interview: read the last 20 interview experiences for that company on GFG and Glassdoor
  • After every interview (pass or fail): write down every question asked within 1 hour. Review after 1 week. Patterns emerge fast
